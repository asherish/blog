# /sync — Bidirectional Translation Sync

Synchronize translations between `articles/` (Japanese, Zenn) and `articles_en/` (English, dev.to).

## Workflow

### Step 1: Detect changes

Run the detection script, passing through any arguments the user provided:

```bash
npx tsx scripts/sync-detect.ts $ARGUMENTS
```

Parse the JSON output to get the sync plan.

- If there are **conflicts**, report them and stop (unless `--prefer` was provided).
- If there are **baselined** slugs, report them (they are now tracked but not translated).
- If there are no **actions**, report "Everything is up to date." and stop.

### Step 2: Translate each article

For each action in the plan:

1. **Read the source file** using the Read tool.
2. **Strip the frontmatter** — do NOT include frontmatter in the translation.
3. **Translate the body content**:
   - For `full-translate`: translate the entire body.
   - For `diff-sync`: translate the entire body (the post-processing will handle the rest).
4. **Write the translated body** to the target file using the Write tool.
   - Write ONLY the translated markdown body — no frontmatter.
5. **Run post-processing**:
   ```bash
   npx tsx scripts/sync-apply.ts <slug> <direction> --title "<translated title>"
   ```
   The `<translated title>` should be the translation of the source article's title.

### Step 3: Report results

After all translations are complete, report a summary:
- Number of articles translated
- List of translated slugs with their directions
- Any skipped or baselined articles

## Translation Rules

You ARE the translator. Follow these rules precisely:

### Language direction
- `ja-to-en`: Translate Japanese → English
- `en-to-ja`: Translate English → Japanese

### What to preserve (do NOT translate)
- Code blocks (``` ```) — keep content exactly as-is
- Inline code (`` ` ``) — keep content exactly as-is
- URLs and file paths
- Command names and CLI arguments
- Emoji characters
- Platform-specific syntax (Zenn `:::` blocks, dev.to `{% %}` tags) — keep as-is, the post-processing script will convert them
- Image references and HTML tags

### Translation style
- Use natural, fluent language appropriate for a technical blog
- Maintain the same tone and level of formality as the source
- Keep technical terms in their commonly accepted form (e.g., don't translate "Docker", "Git", "API")
- For ja-to-en: Use clear, concise English suitable for dev.to audience
- For en-to-ja: Use polite but not overly formal Japanese (です/ます調), suitable for Zenn audience

### Output format
- Output ONLY the translated markdown body
- Do NOT include frontmatter (---) — it will be generated by the post-processing script
- Preserve all markdown formatting (headings, lists, links, etc.)
- Preserve blank lines and document structure
